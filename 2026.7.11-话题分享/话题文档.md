# 第 1 期 AI 分享 · 2026-07-17

> 本期组合：**深读（接住上周）** = 数字分身的司法确权；**新探（额外话题）** = 企微内置 AI Agent「大圆」。

---

## 导语：为什么是这两个话题

上周我们聊"垂直领域数据与 IP 数字分身"时，把分身的能力边界从"初级问答"推到了"数字遗产"。那条线停在**产品能做什么**，但落地前有一道躲不开的关——**分身归谁、侵权怎么判**。这期深读就把这道关补齐。

新探的话题"企微大圆"是这一两周刚内测的产品，它代表另一种范式：AI 不再是一个独立对话框，而是**长在你正在用的工具里、自带上下文的 Agent**。它和我们的数字分身方向是同一股趋势的两个切面。

之所以想专门聊它，是因为我们手头正在做的「撒米工作社区」插件（匿名感谢 / 成长清单）**全部都是落在企业微信里的实现**——自建应用 H5、智能机器人 API 模式、后端定时任务、本地模型识别，一整套自建基建。而大圆这种"嵌入式原生 Agent"的出现，很可能让这类插件需求的**实现方式**在不久后发生迁移：今天要自己搭的"粘合层"，未来可能变成平台原生能力。把它想清楚，直接影响我们 MVP 怎么搭、研发重心放哪。

## 一、本周深读（结合上周）：数字分身的司法确权

### 1.1 为什么现在要谈确权

数字分身不是"画个虚拟人"那么简单。它聚合了多重权益：**形象**（美术作品 / 著作权）、**声音与脸**（肖像权、声音权益 / 人格权）、**驱动它的真人**（中之人 / 表演者权）、**背后的算法与数据**（商业秘密）。任何一层权属不清，产品上线就是埋雷。

上周我们设想"粉丝用垂直数据养出自己的数字分身"，这里立刻冒出三个问题——这恰是本期要回答的。

### 1.2 确权三问

| 问题 | 权益类型 | 归属原则（现行法） |
|------|---------|------------------|
| 形象归谁？ | 著作权（美术作品） | 制作团队；虚拟数字人**本身不是自然人，不享著作权** |
| 声音/脸归谁？ | 人格权（肖像权、声音权益） | 来源于真人→该自然人；未经授权使用即侵权 |
| 中之人归谁？ | 邻接权（表演者权） | 真人驱动型分身的演员，职务表演下可归经营主体 |

### 1.3 实锤判例拆解

#### 判例一：北京互联网法院"天妤 / 之初"虚拟数字人著作权案

- **案号**：(2024)京0491民初4936号 / (2025)京73民终603号
- **入选**：北京高院 2025 年度服务保障新质生产力发展典型案例、2025 北京知识产权保护典型案例（2026.4 发布）
- **事实**：虚拟数字人"天妤"（全平台超 440 万粉丝）、"之初"由四家单位联合制作，某科技（大连）公司为著作权人、北京某科技公司负责运营。合作方前员工孙某某离职后，在模型网擅自售卖两形象模型。
- **裁判要点**：
  - 形象"**不直接来源于真人**"、由制作团队创作、体现对线条/色彩的美学选择与判断 → **构成美术作品**，著作权归制作方；
  - 孙某侵害信息网络传播权，判赔 **1.5 万元**（原告索赔 100 万）；
  - 平台方已尽上传审核、关键词屏蔽、投诉通道等合理注意义务 → **不构成侵权**。
- **对我们有用的规则**：虚拟数字人不是权利主体；权利落在**制作团队**。委托创作按《著作权法》第 19 条——有合同从合同，无合同归受托人。真人驱动型分身背后的"中之人"，在符合条件下享有**表演者权**。

#### 判例二：全国首例"AI 陪伴案"（何某诉自古红蓝公司）

- **地位**：最高人民法院"民法典颁布后人格权司法保护典型民事案例"；北京互联网法院 2021.8 判决、2022 生效
- **事实**：一款记账软件允许用户用他人姓名、肖像自建"AI 陪伴者"，并通过算法推荐、让用户"调教"出亲密互动。公众人物何某被大量用户设为陪伴对象。
- **裁判要点**：
  - **自然人的人格权及于其虚拟形象**。未经同意利用 AI 创设、使用自然人虚拟形象 → 侵害**姓名权、肖像权、一般人格权**；
  - 被告不是"中立技术服务提供者"——它通过规则设定和算法**组织、鼓励**用户生成侵权内容，应作为**内容服务提供者**担责；
  - 判赔礼道歉 + 经济损失 18.3 万 + 精神抚慰金 2 万。
- **一句话**：哪怕形象是 AI 生成的，只要**绑定到特定自然人**，就踩人格权红线。这是做"以真人为原型的数字分身"时第一道合规墙。

#### 判例三：AI 换脸换声带货侵权案

- **时间**：北京互联网法院 2025.8 审结，列入系列涉 AI 侵权典型案例
- **事实**：某文化传媒用教育专家李女士的真实演讲视频 + **AI 合成高度近似的声音**带货图书。
- **裁判要点**：
  - 未经授权使用肖像 + AI 合成声音 → 侵害**肖像权与声音权益**，判赔 **12 万元**；
  - 商家未尽审核注意义务，与发布者承担**连带责任**；
  - 关键规则：**AI 合成声音只要能使公众识别出特定自然人，即落入声音权益保护范围**。
- **关联趋势**：2026.2 施行的《直播电商监督管理办法》已明确要求——用 AI 生成的人物图像/视频做直播电商，**必须显著标识**并向消费者持续提示。

#### 补充锚点（一笔带过，备查）

- **魔珐 Ada 案**（杭州互联网法院，全国首例涉虚拟数字人侵权案，浙江高院 2023 十大典型案例）：真人驱动型虚拟数字人 Ada 构成美术作品；中之人徐某（职务表演）表演者权归公司；判赔 12 万 + 虚假宣传不正当竞争。
- **AI 文生图"吊坠图片"案**（北京互联网法院）：用户主张 AI 文生图享版权，但**无法提供生成过程记录**（提示词、迭代过程）→ 证据不足驳回。规则：**谁主张谁举证**，AI 生成物要证明"智力投入"才算作品。

> [!warning] 给我们的硬提醒
> 如果我们的分身形象用 AI 生成，**务必留存生成记录与创作过程**（提示词、修改轨迹）。否则一旦被人冒用或我们要维权，"吊坠图片案"就是反面教材——主张不了权利。

### 1.4 对我们意味着什么（结合产品）

把三要件落到我们的数字分身产品上：

1. **归属设计**：自研分身 → 著作权默认归公司（制作方），但必须与"中之人"/演员签**授权与收益分配**条款；用**真人肖像做数字分身** → 必须先拿肖像权、姓名权授权，否则直接踩"AI 陪伴案"红线。
2. **授权链路**：对外把分身能力开放给客户/粉丝时，产品里要**内置授权确认 + AI 生成内容标识**，避免"未标识 AI"被判侵权（关联 AI 陪伴案 + 直播电商新规）。
3. **三类高频风险**：
   - 用真人数据训练/驱动分身却未授权；
   - 分身被他人冒用做换脸带货类（判例三）；
   - 用户自建分身侵权，平台是否担责——看你是否"组织/鼓励"内容（判例二的标准）。

### 1.5 一个可讨论框架

**归属矩阵**（谁有什么权）

```
形象著作权      → 制作团队（公司），需与中之人签授权
声音/肖像人格权 → 来源真人，需书面授权
中之人表演者权  → 演员，职务表演可归经营主体
算法/数据       → 商业秘密，归开发方
```

**侵权认定 Checklist**（遇到争议时逐条过）

- [ ] 是否识别到**特定自然人**？（肖像/声音/姓名可对应）
- [ ] 是否**未经授权**使用？
- [ ] 是否**商业性**使用（带货、引流、广告）？
- [ ] 平台/运营方是否**组织、鼓励**了侵权内容？（决定中立还是担责）
- [ ] AI 生成内容是否**显著标识**？

## 二、本周新探（额外话题）：企微内置 AI Agent「大圆」

### 2.1 大圆是什么

企业微信 5.0.9（**2026-06-23**）把内置"智能助理·大圆"推向内测。它的定位很不一样——**长在企业微信工作流里的 AI 助理**，不是一个独立的对话框。

- **交互**：手机端左滑"贴边把手"、电脑端快捷键或左侧功能栏唤起；它会**自动识别你当前所在的界面和上下文**，不用每次把背景重新交代一遍。
- **四大能力**：① 随时唤起；② 总结协作信息（群聊、聊天内容）；③ 读写文档与表格；④ 定时任务（自动整理待办、联网搜索行业动态）。

### 2.2 为什么值得关注

- **体量**：企业微信已接入超 **1400 万家企业**，每天服务的微信用户超 **7.5 亿**。关键在于它与微信互通——这意味着大圆未来不只服务内部协作，还切进**客户经营**场景。
- **双线出牌**：就在大圆内测前三天，微信个人端 AI 助手"小微"刚开启灰度。腾讯在**生活 + 办公**两条 AI 线上几乎同时落子。
- **范式信号**：行业里 2026 年普遍被认为是企业 AI 从"Copilot 辅助"走向"Agent 执行"的拐点。中国企业级 AI 智能体市场 2025 年约 212 亿，预计 2026 年 449 亿、2029 年破 3320 亿（年复合增速约 107%）。

### 2.3 竞争格局（一句话看差异）

| 产品 | 路径 | 侧重 |
|------|------|------|
| 钉钉「悟空」 | 底层 CLI 化，Agent 原生操作上千项能力 | 企业级安全、可控沙箱 |
| 飞书 | "住进数据里"，接入文档/表格/会议 | 结合业务场景问答 |
| 企微「大圆」 | 与微信互通 | **连接**：工作数据 + 客户连接 |

大圆目前聚焦"问答 + 信息整理"，还不会像支付宝/微信助手那样直接替你执行操作；企业数据隐私与安全是它规模化的前提（要读群聊、文档、会议）。

### 2.4 我的分享动机：大圆 × 撒米企微插件

为什么要把"大圆"带进这期分享？因为它和我们正在评估的「撒米工作社区」插件是**同一张考卷的两份答卷**。

我们手头这批插件设计，目标和实现路径已经写得很清楚了：

- **匿名感谢 / 成长清单**：以**企微自建应用（H5 网页）+ FastAPI 后端**承载，零软件成本；
- **待办提醒 / 定时打卡**：必须用**智能机器人 API 模式**（Bot ID + Secret），再配一套**外部 cron 调度器**去扫描、创建企微原生待办；
- **AI 行为识别 / 照片核验**：自接**本地 Ollama + Qwen** 模型，数据不出网；
- **被动全量抓取**（想无感读群聊）：唯一官方合规路是**会话存档 API（450 元/人/年）**，MVP 阶段用"员工主动 @ / 发给应用"的免费半自动路径替代。

一句话概括当前路径：**用"自建应用 + 自建后端 + 自建定时任务 + 自建模型"在企微里拼出一个应用**。每多一个能力，就多一层要自己维护的基建。

而大圆代表的，是反过来的一条路——**能力不再由你从零拼，而是平台原生 Agent 长进工作流里，你只管"吩咐"**。这正是我想借这期分享抛出来的判断：**这些插件需求，未来不久可能有更轻的实现方式。**

### 2.5 大圆作为插件需求的"新实现方式"：能替什么、不能替什么

把大圆已公布的四大能力，对到我们具体的插件需求上，映射关系很直接：

| 我们的插件需求 | 当前实现方式 | 大圆可能带来的新方式 |
|------|------|------|
| 待办提醒 / 定时打卡 | 智能机器人 API 模式 + 外部 cron 调度器 | 大圆**原生定时任务**可直接建待办、整理待办 → 砍掉整套"API 模式 + 自建 cron"基建 |
| "AI 行为记录助手"（员工 @ / 发内容给应用） | @机器人 / 智能机器人 MCP（需逐个授权） | 大圆**自动总结群聊与协作信息**、且自带上下文 → 这条免费半自动路径变成平台原生，无需自建机器人 |
| 成长清单的勾选 / 连续天数 / 看板 | 自建 H5 页面 + 后端积分引擎 | 大圆**读写文档与表格** → 部分交互可变为"Agent 操作文档/表格"，不必全自建 H5 |
| 入口与上下文 | 用户从工作台点开 H5 应用 | 大圆**随时唤起、自动识别当前界面** → Agent 活在上下文里，未必需要单独的"应用" |

> [!note] 一个判断
> 企微原生 Agent 会逐步吃掉"自建应用 + 后端"里那些**粘合层工作**（定时提醒、群聊总结、读写文档、待办）。撒米应该把"哪些能力迟早会被平台原生 Agent 吸收"识别出来，从 MVP 里**剥离 / 借力**，把研发重心放到真正差异化的部分——**匿名机制、积分引擎、垂直社群关系链、AI 识别的精准度**。

**但必须诚实划清边界**（避免把内测产品当银弹）：

- **当前能力有限**：大圆目前聚焦"问答 + 信息整理"，还**不会像支付宝 / 微信助手那样直接替你执行操作**；能力仍在内测，接口与稳定性未定。
- **to-B SaaS 仍需自建后端**：撒米是面向多租户、需 24 小时在线、统一数据存储的产品。大圆（和之前的 WorkBuddy 企微连接一样）是"长在工作流里的个人 / 小团队助手"，**不能当产品的集中式服务端**——要拿它当后端，本质还是自己搭服务器，不如直接做企微自建应用。
- **所以定位是**：MVP / 10 人试点 / 行为激励场景，大圆可能是**近零成本的"AI 行为记录助手"雏形**，用来快速验证闭环；生产级的匿名机制、积分引擎、统一存储，仍要自建。它改变的是"轻量部分怎么搭"，不是"重资产要不要做"。

### 2.6 与我们数字分身方向的关联

我们做的是**文化 / IP 数字分身 + 粉丝精神需求**（主题二 / 三）。大圆代表的那股范式——**Agent 嵌入既有场景、自带上下文，而不是再装一个 App**——很可能也是我们自己数字分身的终局形态之一：

- 分身的价值不只在"长得像"，而在能否**接入真实的关系链与数据流**（企微靠微信连客户，我们靠垂直社群连粉丝）；
- "嵌入式、上下文感知"会比"独立对话机器人"更贴近用户——这对我们设计分身的交互入口是直接启发。

而撒米插件这条线，恰好是同一个判断的**提前演练**：我们今天在企微里为"行为激励"自建的那套基建，未来会有一部分被平台原生 Agent 吸收；我们的数字分身若真要"长在粉丝的工作 / 生活流里"，也得想清楚——哪些能力自己握，哪些借平台的势。

## 三、延伸与行动

- 数字分身**确权是落地必经关**，不是可选项；上周聊的"能力边界"要补上法律这一维。
- 企微大圆印证一个判断：**Agent 正在从"工具"变成"入口"**——谁长在用户的工作/生活流里，谁拿到下一代交互。
